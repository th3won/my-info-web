깃허브 액션 tj-actions/changed-files

공급망 공격 분석 및 대응 방안

23,000개 이상의 저장소가 이번 취약점(CVE-2025-30066)의 영향을 받은 것으로 추정됩니다.

사건 요약

취약점 세부정보

CVE-2025-30066

깃허브 CI/CD 환경에서 널리 사용되는 도구인 tj-actions/changed-files에서 자격 증명 도난에 악용될 수 있는 손상된 사례가 발견되었습니다. 위협 행위자는 액션의 모든 버전(최대 45.0.7)을 수정하여 메모리에서 자격 증명을 스캔하도록 설계된 악성 코드를 삽입했습니다.

8.6
CVSS 심각도 점수
높은 심각도 수준으로 즉각적인 대응이 필요합니다

공격 전략 및 영향

악성 코드 분석

공격자는 액션 코드에 base64로 인코딩된 명령을 삽입했습니다. 이 코드는 깃허브 러너의 메모리를 스캔하여 API 키, 토큰, 비밀번호와 같은 민감한 자격 증명을 찾아내도록 설계되었습니다. 탈취한 데이터는 워크플로 로그에 직접 출력되거나 공격자가 제어하는 깃허브 Gist로 전송되었습니다.

공격 대상

공격은 다음과 같은 민감한 정보를 대상으로 했습니다:

  • AWS 액세스 키
  • 깃허브 개인 액세스 토큰(PAT)
  • npm 토큰
  • 개인 RSA 키
  • 기타 민감한 자격 증명

즉각적인 대응 단계

1단계

감염 여부 확인

깃허브의 코드 검색 기능을 사용하여 조직 내 모든 워크플로에서 tj-actions/changed-files 사용 여부를 검색합니다. 2025년 3월 14일부터 16일 사이에 45.0.7 이하 버전을 사용한 모든 워크플로를 식별합니다.

2단계

로그 분석

실행 로그를 검토하여 의심스러운 IP 주소에 대한 액세스, 비정상적인 아웃바운드 네트워크 요청 또는 "memdump.py"나 base64로 인코딩된 문자열과 같은 키워드를 찾습니다.

3단계

자격 증명 교체

영향을 받은 저장소에서 사용된 모든 중요 자격 증명(API 키, 액세스 토큰, 암호)을 즉시 교체합니다. 해당 기간 동안 실행된 워크플로에서 사용된 모든 비밀은 손상되었을 가능성이 있는 것으로 간주합니다.

4단계

액션 업데이트

tj-actions/changed-files 액션을 최신 패치 버전(>= 46.0.1)으로 업데이트하거나 안전한 다른 액션으로 교체합니다. 보안 강화를 위해 패치된 버전의 특정 커밋 SHA에 고정하는 것을 고려하세요.

장기적인 보안 모범 사례

커밋 SHA에 고정

전체 길이의 커밋 SHA를 사용하여 액션을 고정합니다. 이렇게 하면 불변성을 보장하고 악의적인 업데이트를 방지할 수 있습니다. 수정될 수 있는 태그나 브랜치 이름에 고정하는 것을 피하세요.

보안 효과 높음

허용 목록 기능 활용

명시적으로 허용되고 신뢰할 수 있는 깃허브 액션만 실행하도록 제한합니다. 깃허브 또는 검증된 제작자가 만든 액션 또는 특정 액션만 허용하도록 깃허브를 구성합니다.

보안 효과 높음

런타임 모니터링

보안 거버넌스 구축

보안 정책 개발

GitHub Actions 사용에 대한 명확한 조직 정책을 개발하여, 개발자가 안전하게 액션을 선택하고 구성하는 방법에 대한 지침을 제공합니다. 이 정책에는 다음 사항이 포함되어야 합니다:

  • 승인된 액션 목록 및 평가 기준
  • 새 액션 요청 및 검토 프로세스
  • 액션 사용에 대한 권장 구성 및 제한 사항
  • CI/CD 환경에서의 비밀 관리 지침

개발자 교육

모든 개발자에게 GitHub Actions 보안 모범 사례에 대한 교육을 제공합니다. 이는 보안 인식을 높이고 일상적인 개발 업무에서 안전한 습관을 장려합니다. 특히 다음 내용을 포함하세요:

  • 워크플로 파일에서 안전하지 않은 패턴 식별
  • 타사 액션의 보안 위험 평가 방법
  • 비밀 및 자격 증명 관리 모범 사례
  • 공급망 공격의 징후와 보고 방법

CI/CD 보안은 지속적인 과정입니다. 단일 보안 조치에 의존하지 말고, 다층적인 방어 전략을 구현하세요.

결론 및 종합 권장사항

tj-actions/changed-files 손상 사례는 CI/CD 환경에서 타사 액션의 보안 위험을 강조합니다. 23,000개 이상의 저장소가 영향을 받은 이번 공격은 오픈 소스 공급망의 취약성을 보여줍니다. 이번 사례를 교훈 삼아, 모든 조직은 다음과 같은 종합적인 접근 방식을 취해야 합니다:

  • 단기 대응: 영향을 받은 저장소 식별, 자격 증명 교체, 패치된 버전 업데이트
  • 기술적 보호: 커밋 SHA 고정, 허용 목록 사용, 최소 권한 원칙 적용, OIDC 구현
  • 운영 제어: 정기적인 감사, 지속적인 모니터링, 자동화된 보안 스캔 도구 구현
  • 조직적 접근: 명확한 보안 정책 수립, 개발자 교육, 사고 대응 계획 개발
  • 커뮤니티 협력: 오픈 소스 커뮤니티 내에서 보안 취약점 및 모범 사례 공유

단일 보안 조치가 모든 위험을 방지할 수 없으므로, 다층적인 보안 접근 방식과 지속적인 경계가 필요합니다. 기술적 통제, 인적 요소, 정책 및 절차를 결합한 종합적인 보안 전략을 통해 조직은 미래의 공급망 공격에 대한 복원력을 구축할 수 있습니다.
